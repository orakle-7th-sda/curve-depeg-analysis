본 연구는 Jupiter(Solana 생태계 DEX 어그리게이터) 알고리즘 개선을 목표로 시작했다.
연구 과정에서 알고리즘 개선이 현실적으로 왜 어려운지 네 가지 근거를 도출했다.

---

## 1. 비교 기준점 자체가 없다

알고리즘을 개선하려면 먼저 현재 알고리즘이 어떻게 동작하는지 알아야 한다.
Jupiter의 핵심 라우팅 엔진인 **Metis**는 소스코드가 공개되어 있지 않다.
Jupiter 공식 포럼([Jupiter Metis Routing — Optimizing Swaps on Solana](https://discuss.jup.ag/t/jupiter-metis-routing-optimizing-swaps-on-solana/22152/4))에서
Metis가 Bellman-Ford 알고리즘 기반으로 동작한다는 설명과 성능 지표를 공개하고 있으나,
실제 구현 코드와 세부 파라미터는 제공되지 않는다.

내부적으로 어떤 방식으로 경로를 탐색하는지, 각 풀에 어떤 가중치를 부여하는지,
탐색 깊이는 얼마나 되는지 — 이 모든 것을 외부에서 알 수 없다.

기준점을 모르는 상태에서 "우리 알고리즘이 더 낫다"는 주장은 성립하지 않는다.

---

## 2. 실험 결과를 신뢰할 수 없다

기준점 문제를 우회해 실제 Jupiter와 결과를 비교하는 실험을 설계할 수 있다.
그러나 이 실험도 근본적인 문제를 가진다. **결과 차이의 원인을 특정할 수 없다.**

어그리게이터는 거래를 실행하기 직전, 여러 DEX에 현재 가격을 물어본다.
이 조회 시점과 실제 거래 체결 시점 사이에 수많은 변수가 개입한다.

- **풀 상태 변동**: 다른 사용자의 거래가 끼어들어 풀 잔고를 바꿔놓는다. Solana는 400ms마다 블록이 생성되고, 그 사이 수백 건의 트랜잭션이 처리된다.
- **네트워크 혼잡**: 같은 풀이라도 RPC 노드 응답 속도가 매번 다르다. 혼잡한 순간에는 일부 견적이 이미 낡은 상태로 도착하기도 한다.
- **MEV 봇 개입**: 대형 거래가 감지되면 봇이 먼저 스왑을 실행해(프론트러닝) 내 거래 전 풀 상태를 바꿔놓는다. 알고리즘과 무관하게 결과가 나빠진다.
- **슬리피지 허용치**: 비교 대상과 허용 범위 설정이 다르면, 같은 경로라도 체결 조건이 달라진다.

이 변수들은 통제할 수 없다. 결과 차이가 알고리즘 때문인지, 그 순간의 환경 때문인지 구분할 방법이 없다.

> **시연 환경 실행 결과** (`aggregator_benchmark.py`, 벤치마크 2):
>
> ```
> Same algorithm / same input / $80,000 large trade injected mid-run
>
> Run 1:  9,999.0 USDT  (selected pool: Saber)
> Run 2:  9,995.0 USDT  (selected pool: Orca)   ← pool changed too
> Diff:       3.99 USDT  (0.04%)
> ```
>
> 알고리즘은 동일했다. 차이는 중간에 발생한 풀 상태 변동 때문이다.
> 실제 환경에서는 이 변동이 항상 존재하며, 통제할 수 없다.

---

## 3. 알고리즘 개선보다 다른 접근이 더 효과적일 수 있다 — 그러나 검증할 수 없다

어그리게이터의 동작을 단계별로 분해하면 다음과 같다.

1. 여러 DEX에 현재 가격 조회
2. 조회 결과를 취합
3. 수신된 가격들을 비교해 최적 경로 선택 ← **알고리즘이 작동하는 구간**
4. 선택된 경로로 거래 실행

실제 Jupiter가 1~2번을 어떻게 처리하는지 — 실시간 RPC 조회인지, 사전 인덱싱인지, gRPC 스트리밍인지 — 는 소스가 공개되지 않아 외부에서 알 수 없다.

> **시연 환경 실행 결과** (`aggregator_benchmark.py`, 벤치마크 1):
>
> *(아래 수치는 풀당 200ms의 실시간 RPC 조회를 가정한 시뮬레이션 환경 기준이다.*
> *실제 Jupiter의 내부 아키텍처는 다를 수 있다.)*
>
> ```
> Pool price queries (6 pools, real-time RPC assumed):  1,219 ms
> Path-finding algorithm (CPU compute):                    0.05 ms
> ```
>
> 이 가정이 성립한다면 알고리즘 계산이 전체의 극히 일부를 차지한다.
> 알고리즘을 2배 빠르게 개선해도 전체 응답 시간은 거의 줄지 않는다.

> **Jupiter 공식 사례**:
>
> Jupiter 개발팀이 quote latency를 200ms → 10ms로 줄인 방법은 알고리즘 개선이 아니었다.
> 조회하는 풀의 수를 줄이는 것(skipping multi-route aggregation)이 핵심이었다.
> ([Jupiter Developers Updates, 2025.10](https://dev.jup.ag/updates))
>
> *(단, 해당 최적화는 신규 상장 토큰 마켓(pre-graduated bonding curve)에 한정된 사례다.)*
>
> 이는 **알고리즘이 아닌 다른 접근**(조회 대상 축소)이 더 큰 성능 개선을 가져올 수 있음을 보여주는 공식 사례다.
> 정확한 병목이 어디인지 외부에서 특정할 수 없는 만큼, 알고리즘 개선이 전체 성능에 얼마나 기여할지도 예측할 수 없다.

---

## 4. 이미 성숙한 시스템이다

Jupiter는 2021년경 Solana 생태계에서 서비스를 시작해 수억 달러 규모의 거래를 처리해왔다.
수년간 실거래 데이터를 기반으로 지속적으로 튜닝된 시스템이다.

공식 문서에 따르면 Metis는 Bellman-Ford를 "heavily modified"한 변형이며,
이전 버전(v2) 대비 평균 5.22% 더 나은 가격을 제시한다고 명시되어 있다.
그러나 어떻게 변형했는지 — 가중치 설정, 탐색 범위 제한, 수렴 조건 —
는 공개되지 않는다. 결국 1번 문제로 다시 돌아온다.
**변형 방식을 모르면 동일 조건으로 비교하는 것 자체가 불가능하다.**

2025년 말 기준 Jupiter의 평균 슬리피지는 타 어그리게이터 대비 15~20% 낮은 수준으로
관찰된다(Jupiter 분석 보고서 기준). 이는 이미 상당한 최적화가 이루어진 결과다.

단기 연구로 이 시스템 대비 유의미한 알고리즘 개선을 도출하는 것은 현실적으로 어렵다.

---

## 결론

네 가지 이유를 종합하면, 알고리즘 개선 방향의 연구는 다음 문제를 동시에 안고 있다.

| 문제           | 내용                                       |
| -------------- | ------------------------------------------ |
| 비교 불가      | 소스 미공개로 베이스라인 없음              |
| 검증 불가      | 환경 변수와 알고리즘 효과를 분리할 수 없음 |
| 효과 불명      | 내부 아키텍처 미공개 → 병목 특정 불가, 알고리즘 개선 효과 예측 불가 |
| 개선 여지 부족 | 이미 성숙한 시스템                         |
